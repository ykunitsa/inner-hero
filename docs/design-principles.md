# Внутренний герой — принципы интерфейса

Экраны этого приложения проектируются для человека, которому может быть плохо прямо
сейчас. Это главный дизайн-констрейнт: не «красиво и современно», а «очевидно, что
нажать, когда сил нет». Документ дополняет продуктовые принципы `docs/redesign-spec.md` §1
и обязателен при проектировании (`/design-screen`) и проверке (`/design-review`) экранов.

---

## 1. Один экран — одна работа

У экрана есть одна цель и одно главное действие. Если на макете два равнозначных
акцента — это два экрана или лишний элемент. Акцентная карточка/кнопка — максимум
одна; всё остальное визуально отступает (вторичные стили, `TextColors.secondary`).

## 2. Выбор — дорогая операция

- Умные дефолты: подставляется последнее использованное значение; «сменить» — отдельное
  тихое действие, а не пикер по умолчанию.
- Чипсы вместо клавиатуры, слайдеры и сегменты вместо ввода цифр, «да/нет» вместо шкал,
  где хватает «да/нет».
- Мерило формы: заполняется большим пальцем одной руки за ~30 секунд. Каждое поле,
  требующее клавиатуры, должно оправдаться.
- Если пользователь не может выбирать (плохое состояние) — выбор не предлагается вовсе
  (спека 1.2).

## 3. Иерархия и тишина

- Один заголовок уровня экрана; типографика только из шкалы `AppTextStyle`.
- Ритм отступов только из `Spacing`, радиусы из `CornerRadius` — никаких магических чисел.
- Проверка на прищур: при расфокусе экрана иерархия должна читаться за секунду —
  что здесь главное, что второстепенное.
- Пустое место — инструмент, а не недоработка. Не заполнять «полезными советами».

## 4. Состояния проектируются сразу

Для каждого экрана до кода описываются: пустое состояние (в т.ч. правило
`sessions == 0` — подпись-объяснение и карточка статьи до первой сессии), состояние
«ничего не запланировано» (тихая строка, НЕ карточка-призыв), ошибка, выход.
Выход из флоу — это данные, а не отмена (спека 1.5): кнопка называет факт
(«Закончить раньше»), а не «Отменить».

## 5. Тон текстов

- Спокойные факты без оценок: «Записать», «Готово», «16 июля · 6 из 7» — не
  «Отлично!», «Так держать!», «Ты справился!».
- Никакого стыда и упрёков: не «Вы пропустили 3 дня», а просто актуальное состояние.
- Кнопки — глаголы действия; подписи — состояние, не маркетинг (спека 1.3).
- Строки в коде — английский источник + русский перевод в `.xcstrings`; обе версии
  проверяются на тон.

## 6. Доступность — норма, не фича

- Dynamic Type: макет не ломается на крупных размерах; `fixedSize(horizontal: false,
  vertical: true)` для многострочных текстов.
- VoiceOver: осмысленные `accessibilityLabel`, объединение составных элементов,
  `accessibilityHidden(true)` для иконок-украшений.
- Цели касания ≥ 44pt (`TouchTarget.minimum`), контраст в обеих темах (токены DS
  это гарантируют — ещё одна причина не хардкодить цвета).
- `Reduce Motion` уважается для всех появлений/переходов.

## 7. Движение и хаптика

- Анимации только из `AppAnimation`, тонкие и быстрые; появление списка — да,
  салют — никогда.
- Хаптика функциональная (подтверждение сохранения, фаза таймера), не праздничная.
  В контексте тревоги вибрация — сигнал, к которому пользователь прислушивается:
  не тратить его на украшательство.

## 8. Чего в этом интерфейсе не бывает

Бейджи и счётчики непрочитанного; стрики и серии; прогресс-кольца по дням; проценты
выполнения недели; конфетти и поздравительные модалки; «мотивирующие» цитаты;
красные предупреждающие цифры; пуши-упрёки. Если элемент существует, чтобы
«вовлекать», а не помогать сделать назначенное, — его не должно быть (спека 1.4).

## Чеклист ревью экрана

1. Иерархия читается за секунду? Главное действие одно и очевидно?
2. Сколько тапов до результата на «плохом дне»? Что можно убрать?
3. Все значения из токенов DS? (`grep` по `.system(size:`, `Color(`, числовым паддингам)
4. Состояния: пустое / `sessions == 0` / ошибка / выход-с-данными — есть?
5. Тексты: тон спокойный, без оценок и упрёков? EN-источник + RU-перевод?
6. Dynamic Type / VoiceOver / 44pt / тёмная тема — проверены?
7. Ничего из раздела 8 не пролезло?
