# 11.7 — iOS widgets + the one external entry point

Status: implemented in code (July 2026). Not yet run on a device; the target and the
App Group still have to be created by hand in Xcode (see "Xcode setup" below), so the
extension does not build until that is done.

The app-side plumbing — the snapshot, the deep link, the notification payloads, the
extracted recurrence rule — builds and ships in the app target today and is covered
by tests. The `InnerHeroWidget/` sources compile only once the target exists.

## What was built

Four widgets, not one. Spec §9 was titled "один"; §3 separately describes a permanent
"Записать экспозицию" button. The two contradicted each other, and the priority
widget resolved the contradiction the wrong way — it demotes the exposure button to
third place, hidden for days whenever a BA tail is open, even though the situational
form is the one entry with a shelf life (§1.6: captured while fresh, never
reconstructed).

- **Today** (`systemSmall` + `systemMedium`) — the §9 priority, whole and unweakened:
  open BA tail → next scheduled item within 24h → log-exposure fallback. Medium exists
  so a long activity title is not clipped, never to show a second thing.
- **Log an exposure** (`systemSmall`) — §3's button. One state, always the same, reads
  nothing. Works with the app never opened and under App Lock.
- **Breathing**, **Relaxation** (`systemSmall`) — the exercise's door + its ladder
  position, or the corrective phrase while `sessions == 0` (§1.7).

BA gets no widget of its own: its door is the "how much energy" question, so a BA
button would open a question rather than an action (§1.2). What BA puts on a home
screen is its tail, which lives in the Today widget.

### Decisions

1. **Snapshot, not a shared store.** The widget reads a flat `WidgetSnapshot` (JSON in
   the App Group), not SwiftData. Moving the store into the group container would move
   the file (losing the data) and compile the schema into the extension, to answer
   "what is next" with three strings. `WidgetSnapshotBuilder` builds it (pure), the app
   writes it on background/active, then `reloadAllTimelines()`.

2. **Raw recurrence, not "today's list".** The schedule travels as its recurrence
   fields; the widget runs `ScheduleRecurrenceRule` itself and stays truthful for days
   without the app. That rule was extracted out of `ScheduleItem` — the model now
   delegates and owns no occurrence logic. The old tests still call it through
   `ScheduleItem` on purpose (the extraction changed nothing).

3. **Ladder positions, not dated subtitles.** The launcher subtitle "7 groups · today"
   becomes a lie at midnight, and a widget cannot recompute it. The snapshot carries
   the position alone ("7 groups"); the day is History's job. Exposure has no position
   (its ladder is a ratio over a window — a statement about the log, not a place to
   stand) so its widget shows none.

4. **App Lock redacts on write.** With the lock on, the snapshot is built empty of
   titles, times and positions. Redaction happens in the builder, so the shared
   container never holds what the lock hides — a widget that received the strings and
   declined to draw them would still be leaking them to anything that can read the
   group (§1.9).

5. **One deep link, `innerhero://`.** `DeepLink` names a *door*, never a step inside a
   flow. Widgets and the schedule/BA reminders all route through it; the
   planned-exposure end signal deliberately carries none (its "after" form only exists
   inside a session the app is no longer running — a link there would invent data,
   §1.6). `DeepLinkInbox` parks a link until a screen can honour it: a cold launch has
   no tab bar yet, and App Lock is on screen but must open nothing. `TodayView` spends
   it, and never over an existing flow (obeying a tap by discarding a mid-flight
   session would lose data, §1.5).

6. **`WidgetTile` is a new component.** `ExerciseRow`/`HeroFeatureCard` are full-width
   `Button`s; a button inside a widget cannot exist, and their chrome fills 158pt on
   its own. The tile reuses every token and the same glyph-over-labels hierarchy, and
   lives in `InnerHeroWidget/` because it has one host.

## Xcode setup

The target `InnerHeroWidgetExtension` was created (July 2026, bundle id
`wrongteam.Inner-Hero.InnerHeroWidget`). Its `InnerHeroWidget/` folder is a
synchronized root group, so every source in it is a target member automatically. Done
in the pbxproj already, no UI needed:

- Xcode's template sources (`AppIntent`, `InnerHeroWidget`, `…Control`,
  `…LiveActivity`) deleted; `InnerHeroWidgetBundle.swift` restored to the four real
  widgets. The template `Info.plist` and `Assets.xcassets` (which carry the
  `AccentColor`/`WidgetBackground` the build settings name) are kept.
- The nine shared sources are added to the widget target via a
  `PBXFileSystemSynchronizedBuildFileExceptionSet` on the **`Inner Hero`** synchronized
  group (an exception set whose `target` is the *other* target means "also builds
  here"): `Core/Widget/{WidgetSnapshot,WidgetState}.swift`,
  `Core/Navigation/DeepLink.swift`, `Models/{ScheduledExercise,ScheduleRecurrenceRule}.swift`,
  `Core/DesignSystem/{DesignSystem,Typography}.swift`, `Resources/Localizable.xcstrings`,
  `Resources/Assets.xcassets`. `ViewModifiers.swift` is **not** needed — `WidgetTile`
  uses only tokens and `.appFont`. The widget target builds green.
- Widget version aligned to the app (`MARKETING_VERSION = 2`,
  `CURRENT_PROJECT_VERSION = 2.0.1`) — an extension whose CFBundleVersion differs from
  its host is an App Store reject.

**The one step still requiring the UI** (a developer-portal round trip, so it can't be
scripted): add capability **App Groups** → `group.wrongteam.Inner-Hero` on **both** the
`Inner Hero` and `InnerHeroWidgetExtension` targets, in Signing & Capabilities. Until
then `UserDefaults(suiteName:)` returns nil on both sides — the app writes the snapshot
nowhere and every widget falls back to its default state (it still builds and runs).
